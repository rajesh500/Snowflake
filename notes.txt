
Snowflake architecture:
Snowflake’s architecture is a unique hybrid of traditional shared-disk and shared-nothing database architectures, designed specifically for the cloud. It separates storage, compute, and cloud services into three distinct, independently scalable layers.

Database Storage (Storage Layer): AWS S3, BLOB, GCP
Query Processing (Compute Layer): virtual warehouses (MPP compute cluster)
Cloud Services (Brain Layer): manages authentication, infrastructure, metadata, security, and query optimization.

Standard and Snowpark-Optimized:
Standard for SQL
Snowpark-Optimized for snowpark.

Warehouse Sizing (T-Shirt Sizes): X-Small to 6X-Large.

Database types:
Standard (Permanent) Databases: full data protection including time travel upto 90 days and Fail-safe.
Transient Databases: no permanent storage (kind of staging), time travel 1 day, no fail-safe.
Temporary: Session base, once session end will drop automatically.

Slow Query Performance:
When you notice that a query is running slowly, check whether an overloaded warehouse is causing the query to compete for resources or get queued:
•	If the running query load is high or there’s queuing, consider starting a separate warehouse and moving queued queries to that warehouse. Alternatively, if you are using multi-cluster warehouses, you could change your multi-cluster settings to add additional clusters to handle higher concurrency going forward.
•	If the running query load is low and query performance is slow, you could resize the warehouse to provide more compute resources. You would need to restart the query once all the new resources were fully provisioned to take advantage of the added resources.


Time-Travel:
Snowflake Time Travel is a continuous data protection (CDP) feature that allows users to access, query, and restore historical data that has been changed or deleted within a defined retention period.

Core Capabilities:
Query Historical Data
Restore Dropped Objects
Cloning Past States

Ex:
Consider Table-A and its cloned snapshot of Table-B.
MP-1, MP-2, MP-3 pointing to same micro partition.

Micro-partitions in Snowflake are immutable i.e., once created lasts in the same state until table is dropped. Hence, for any change in the data of a micro-partition, a new micro-partition is created with updated changes and metadata will point to the newly created micro partition. The older partition is retained for time-travel and Fail-safe purposes
