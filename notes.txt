
Snowflake architecture:
Snowflake’s architecture is a unique hybrid of traditional shared-disk and shared-nothing database architectures, designed specifically for the cloud. It separates storage, compute, and cloud services into three distinct, independently scalable layers.

Database Storage (Storage Layer): AWS S3, BLOB, GCP
Query Processing (Compute Layer): virtual warehouses (MPP compute cluster)
Cloud Services (Brain Layer): manages authentication, infrastructure, metadata, security, and query optimization.

Standard and Snowpark-Optimized:
Standard for SQL
Snowpark-Optimized for snowpark.

Warehouse Sizing (T-Shirt Sizes): X-Small to 6X-Large.

Database types:
Standard (Permanent) Databases: full data protection including time travel upto 90 days and Fail-safe.
Transient Databases: no permanent storage (kind of staging), time travel 1 day, no fail-safe.
Temporary: Session base, once session end will drop automatically.

Slow Query Performance:
When you notice that a query is running slowly, check whether an overloaded warehouse is causing the query to compete for resources or get queued:
•	If the running query load is high or there’s queuing, consider starting a separate warehouse and moving queued queries to that warehouse. Alternatively, if you are using multi-cluster warehouses, you could change your multi-cluster settings to add additional clusters to handle higher concurrency going forward.
•	If the running query load is low and query performance is slow, you could resize the warehouse to provide more compute resources. You would need to restart the query once all the new resources were fully provisioned to take advantage of the added resources.


Time-Travel:
Snowflake Time Travel is a continuous data protection (CDP) feature that allows users to access, query, and restore historical data that has been changed or deleted within a defined retention period.


Fail-safe:
7 days after Time Travel ends, Snowflake keeps a read-only copy internally for disaster recovery.

Fail-safe: Snowflake can restore the table if something went catastrophically wrong.

Micro-partitions in Snowflake are immutable i.e., once created lasts in the same state until table is dropped. Hence, for any change in the data of a micro-partition, a new micro-partition is created with updated changes and metadata will point to the newly created micro partition. The older partition is retained for time-travel and Fail-safe purposes

Dynamic table:
It stores the results of a query, but unlike a regular table, it updates automatically as the underlying source data changes.
Think of it as a self-updating, incrementally refreshed table.


CREATE DYNAMIC TABLE daily_sales
AS
SELECT
    DATE(order_date) AS day,
    SUM(amount) AS total_sales
FROM orders
GROUP BY DATE(order_date);

As new rows are inserted into orders, Snowflake automatically updates daily_sales.
No manual INSERT or UPDATE is needed.

External Tables:
In Snowflake, an external table lets you query data stored outside Snowflake (like in cloud storage) without loading it into Snowflake tables.

Data lied in the cloud storage(S3, ADLS, GCP) a stage going to be created in snowflake and an external table is created.

CREATE STAGE my_stage  URL = 's3://my-bucket/data/';
CREATE EXTERNAL TABLE ext_sales ( order_id STRING AS (value:order_id::STRING), amount NUMBER AS (value:amount::NUMBER))

stream:
In Snowflake, a stream is used to track changes (CDC – Change Data Capture) in a table—like inserts, updates, and deletes.

CREATE OR REPLACE TABLE sales (id INT, amount NUMBER);
CREATE OR REPLACE STREAM sales_stream ON TABLE sales;   -- now it kind of streaming data into table with CDC.
CREATE STREAM s2 ON TABLE sales APPEND_ONLY = TRUE;   -- Append-Only Stream (track only inserts).
CREATE STREAM s3 ON VIEW my_view;    -- Tracks changes from underlying tables.
 
Raw Table (landing) -> Stream (CDC) -> Task (scheduled or triggered) -> Clean Table (analytics-ready).

Why Snowflake over other Warehouses:
Option to choose the cloud provider instead of tightly binding to one of them (AWS/GCP/Azure)
Provides an efficient serverless architecture
Queries are stored in the cache for 24hrs saving compute time and resources
Time-travel feature to revert to query IDs (starts 24hrs for standard account upto 90 days with Enterprise version)
Decouples Compute and Storage and saves costs with columnar compression and other techniques.
Scales efficiently unlike other solutions like Redshift, which scales by compute nodes, charging per hour.
Doesn't charge for data ingress and metadata storage.
Provides in-house visualization tools to understand the results while querying.

External Table      vs     Iceberg table:
external table  query files in place(read only).
Lets you query data directly from files (S3, Azure, GCS)
Data stays in files (CSV, Parquet, JSON)

Key characteristics
❌ Read-only (no INSERT/UPDATE)
❌ No data stored in Snowflake


Iceberg table   Full table stored in your data lake (read/write).
A full table format (like normal Snowflake tables)
But data is stored in your cloud storage
Uses Apache Iceberg format
👉 Key characteristics
✅ Read + Write (INSERT, UPDATE, DELETE)
✅ ACID transactions
✅ Schema evolution
✅ Can be used by multiple engines (Spark, Snowflake)

when to use it:
You need full table operations (INSERT/UPDATE)
You want data lake + warehouse together

create external volume + Iceberg table.


Iceberg:
A table whose data is stored in your cloud storage (like S3) using the Apache Iceberg format, but can be queried and managed from Snowflake.
Snowflake (compute + metadata)  -->  External Volume   -->  S3 / Azure / GCS (Iceberg data files).

underneath iceberg uses parquet file format.

Feature:
External storage.
ACID transactions
Schema evolution (add/drop column)
time travel

snapshot:
Snapshots are NOT deleted automatically right away.
They are removed only when you run a cleanup process (like vacuum/expire) and they are older than the retention period.

Iceberg does not overwrite data, Every change creates a new snapshot.
Snapshot 1 → Initial load
Snapshot 2 → Insert new data
Snapshot 3 → Update records
Snapshot 4 → Delete records

Iceberg:

Writes new Parquet file
Creates new snapshot
Updates metadata pointer


Time travel:
it will go that specific version instead of latest version.



✅ ACID transactions   Safe updates
✅ Time travel        Query past data
✅ Rollback          Revert to old snapshot
✅ Incremental processing   Process only new data
